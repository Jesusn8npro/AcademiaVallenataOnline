### Características

| Feature | Detalle |
|---|---|
| Timeline visual | Bandas de color por sección, zona verde para extensión de notas |
| Seek bar | Input range invisible superpuesto al visual (drag para buscar) |

### Props interface

```ts

  duracionAudio?: number;
```

### Dos modos de sincronización del slider

- **Modal (sliderRef pasado):** El RAF loop interno del modal actualiza `sliderRef.current.value` directamente — muy performante.
- **Hero (sin sliderRef):** Un `useEffect` sobre `tickLocal` actualiza el valor del slider cuando la prop cambia.

## 3. Arquitectura Portal — Modal proyecta el transport al footer

### Problema que resuelve
El modal editor tiene su propio motor (RAF loop + audio element propio). Si el transport fuese un componente externo, no tendría acceso a ese estado interno. La solución es que el modal renderice `BarraTimelineProMax` **dentro de su árbol** pero lo proyecte físicamente al footer con un portal.

### Cómo funciona

**En `EstudioAdmin.tsx`** — se define el slot del portal en el mismo lugar donde antes estaba el reproductor:

```tsx
{/* Slot vacío cuando el modal está abierto — el portal lo rellena */}
{rec.cancionEnModalEditor && (
  <div id="barra-timeline-slot" className="estudio-practica-libre-transport-fixed" />
)}

{/* Transport hero — solo cuando NO hay modal */}
{!rec.cancionEnModalEditor && (hero.reproduciendo || rec.hayGrabacionActiva || Boolean(libreria.pistaUrl)) && (
  <div className="estudio-practica-libre-transport-fixed">
    <BarraTimelineProMax ... />
  </div>
)}
```

**En `ModalEditorSecuencia.tsx`** — el portal se registra después del mount:

```tsx
const [portalTarget, setPortalTarget] = useState<HTMLElement | null>(null);
useEffect(() => { setPortalTarget(document.getElementById('barra-timeline-slot')); }, []);

return (
  <>
    <div className="editor-secuencia-modal">
      {/* ... contenido del modal ... */}
    </div>

    {portalTarget && createPortal(
      <BarraTimelineProMax
        secciones={sec.secciones}
        totalTicksModal={totalTicksModal}
        bpmModal={bpmModal}
        bpmOriginal={cancion?.bpm || 120}
        setBpmModal={setBpmModal}
        onCambiarBpm={onCambiarBpm}
        punchInTickLocal={punch.punchInTickLocal}
        punchOutTickLocal={punch.punchOutTickLocal}
        secuenciaEditada={secuenciaEditada}
        sliderRef={sliderRef}
        isSeekingRef={isSeekingRef}
        tickLocal={tickLocal}
        tiempoAudioActual={tiempoAudioActual}
        duracionAudio={duracionAudio}
        reproduciendoLocal={reproduciendoLocal}
        handleSeek={handleSeek}
        handleReset={handleReset}
        saltarSegundos={saltarSegundos}
        togglePlay={togglePlay}
      />,
      portalTarget
    )}
  </>
);
```

### Por qué usar `Fragment` (`<>`)
React requiere un único elemento raíz por return. Al tener el `<div>` del modal Y el portal como hermanos, se necesita el Fragment `<>...</>`.

### Exclusión mutua garantizada
- `#barra-timeline-slot` solo existe cuando `rec.cancionEnModalEditor !== null`.
- El transport hero solo se renderiza cuando `!rec.cancionEnModalEditor`.
- Nunca hay dos barras simultaneas.

## 7. Bug corregido — "Siempre suena la misma pista"

**Archivo:** `src/Paginas/AcordeonProMax/Admin/Hooks/useEditorSecuenciaAdmin.ts` línea 292

**Problema:** Al reproducir canciones desde el panel de librería, siempre sonaba el mismo audio (el de la última canción "preparada"), sin importar qué canción se seleccionara.

**Causa:** `handleReproducirLibreria` llamaba a `onReproducirSecuencia` directamente sin actualizar `pistaUrl` ni el BPM. El hook `useAudioFondoPracticaLibre` que maneja el audio hero nunca recibía la nueva URL.

**Fix:**
```ts
// ANTES
const handleReproducirLibreria = useCallback((cancion: any) => {
  const cancionPreparada = { ...cancion, secuencia_hero: ... };
  onBuscarTick(0);
  onReproducirSecuencia(cancionPreparada);
}, [cancionEditandoSecuencia?.id, onBuscarTick, onReproducirSecuencia]);

// DESPUÉS
const handleReproducirLibreria = useCallback((cancion: any) => {
  const cancionPreparada = { ...cancion, secuencia_hero: ... };
  prepararCancionEnEscenario(cancionPreparada);   // ← actualiza pistaUrl, bpm, tonalidad
  onBuscarTick(0);
  onReproducirSecuencia(cancionPreparada);
}, [cancionEditandoSecuencia?.id, onBuscarTick, onReproducirSecuencia, prepararCancionEnEscenario]);
```

**Qué hace `prepararCancionEnEscenario`:**
```ts
setPistaUrl(cancionPreparada?.audio_fondo_url || null);  // actualiza URL para useAudioFondoPracticaLibre
onCambiarBpm(bpmCancion);                                // sincroniza BPM
setBpmHero(bpmCancion);
setCancionActivaLibreria(cancionPreparada);
// También sincroniza tonalidad si cambió
```

## 8. Bug corregido — Sync loop de audio se moría al pausar

**Archivo:** `src/Paginas/AcordeonProMax/Admin/Componentes/ModalEditorSecuencia.tsx` línea 180

**Problema:** Si el audio se pausaba momentáneamente (tab switching, latencia del browser), el loop de sincronización moría y el audio quedaba desincronizado o "trabado" al volver.

**Causa:** El `return` sin `requestAnimationFrame` dentro del loop lo mataba permanentemente.

**Fix:**
```ts
// ANTES — mataba el loop si audio estaba pausado
const sync = () => {
  if (!audioRef.current || audioRef.current.paused) return;  // ← loop muerto
  // ... sync ...
  syncAudioRef.current = requestAnimationFrame(sync);
};

// DESPUÉS — el loop siempre vive, solo skip del sync cuando pausado
const sync = () => {
  if (audioRef.current && !audioRef.current.paused) {
    // ... sync ...
  }
  syncAudioRef.current = requestAnimationFrame(sync);  // ← siempre continúa
};
```

### Bug 409 Conflict — RPC `SECURITY DEFINER`

Al borrar, Supabase devolvía 409. La causa real (descubierta tras varios intentos):
- Tablas dependientes (`scores_hero`, `xp_cancion_usuario`, `monedas_cancion_usuario`) **no tienen política RLS DELETE** para el rol `authenticated`.
- Aunque la FK tenga `ON DELETE CASCADE`, la cascada se ejecuta bajo la sesión del usuario y RLS bloquea silenciosamente → FK violation → 409.

**Solución:** función Postgres `eliminar_canciones_hero(p_ids uuid[])` con `SECURITY DEFINER` que valida `rol='admin'` y limpia en orden:
1. `scores_hero` (DELETE)
2. `xp_cancion_usuario` (DELETE — CASCADE no funcionaba por RLS)
3. `monedas_cancion_usuario` (DELETE)
4. `favoritos_acordeon_hero` (DELETE)
5. `grabaciones_estudiantes_hero` → `cancion_id = NULL` (preserva grabaciones del estudiante)
6. `pistas_acompanamiento` → `cancion_hero_id = NULL`
7. `canciones_hero` (DELETE final)

Migración aplicada en Supabase project `tbijzvtyyewhtwgakgka`: `eliminar_canciones_hero_rpc`.

```ts
// cancionesHeroService.ts
export async function eliminarCancionesHero(ids: string[]) {
  if (!ids.length) return;
  const { error } = await supabase.rpc('eliminar_canciones_hero' as any, { p_ids: ids });
  if (error) throw error;
}
```

## 12. Bug — Primera reproducción del transport requería Reset

**Archivo:** `src/Paginas/AcordeonProMax/Admin/Hooks/useEditorSecuenciaAdmin.ts` líneas 274-290.

**Causa:** `reproducirCancionActivaDesdeTick` usaba `setTimeout(0)` anidados con orden indeterminista. `reproducirSecuencia` (en `useReproductorHero.ts:252`) resetea `tickRef = 0` internamente, pero el segundo `setTimeout` programado para `onBuscarTick` podía ejecutarse antes/después → reloj mal sincronizado en el primer Play.

**Fix:** llamar todo síncronamente, en el orden que ya funcionaba en `handleReproducirLibreria`:

```ts
// ANTES
window.setTimeout(() => {
  onReproducirSecuencia(cancionBase);
  window.setTimeout(() => onBuscarTick(...), 0);
}, 0);

// DESPUÉS
prepararCancionEnEscenario(cancionBase);
onReproducirSecuencia(cancionBase);
if (tickInicio > 0) onBuscarTick(Math.max(0, Math.floor(tickInicio)));
```

`reproducirSecuencia` ya resetea a 0 internamente, así que `onBuscarTick` solo se llama si `tickInicio > 0`.

## 15. Cuantización del primer ataque al beat

**Archivo:** `src/Core/hooks/useReproductorHero.ts` (después de `seqOrdenada`).

**Problema:** una grabación con `usoMetronomo=true` cuyo primer tick estaba off-beat (ej: 633.86 a 120 BPM = beat 3.30) sonaba "desincronizada" con los clicks del metrónomo (que caen en beats exactos 192, 384, 576, 768...).

**Fix:** al cargar la secuencia, detectar el offset del primer tick respecto al beat más cercano y desplazar **toda la secuencia** por ese offset (preserva el feel interno):

```ts
if ((cancion as any).usoMetronomo && seqOrdenada.length > 0) {
    const resolucion = (cancion as any).resolucion || 192;
    const primerTick = seqOrdenada[0].tick;
    const beatMasCercano = Math.round(primerTick / resolucion) * resolucion;
    const offset = primerTick - beatMasCercano;
    if (Math.abs(offset) > 2 && Math.abs(offset) < resolucion) {
        seqOrdenada = seqOrdenada.map(n => ({ ...n, tick: n.tick - offset }));
    }
}
```

Esto NO modifica la BD; solo afecta la reproducción.

## 21. Bug crítico — Drift de metrónomo paralelo. Sustitución por `onBeat` síncrono

**Archivos:**
- `src/Paginas/AcordeonProMax/Hooks/useLogicaProMax.ts`
- `src/Paginas/AcordeonProMax/Admin/Hooks/useEstudioAdmin.ts`

**Problema:** después de un rato bajando BPM con el slider, el metrónomo se desfasaba progresivamente respecto a las notas y el MP3.

1. `useReproductorHero` — usa `motorAudioPro.tiempoActual`. Re-checkpoint con ATOMIC SYNC al cambiar BPM.

## 22. Drift MP3 vs secuencia — re-checkpoint y micro-corrección continua

**Archivo:** `src/Paginas/AcordeonProMax/PracticaLibre/Hooks/useAudioFondoPracticaLibre.ts`

**Problema:** tras §21 los clicks quedaban perfectos con las notas, pero el **MP3** seguía sufriendo drift contra el motor de ticks. El usuario reportaba que tras minutos bajando BPM, el MP3 iba "por un lado" y la secuencia "por otro".

**Causa:**
- `useReproductorHero` re-checkpointea con ATOMIC SYNC al cambiar BPM (usando `AudioContext.currentTime`).
- `HTMLAudioElement.currentTime` tiene su propio reloj y solo cambiaba `playbackRate` sin re-checkpointar.
- La micro-sincronización vieja tenía threshold `0.15s` con cooldown `2s` — demasiado tolerante.

**Fix 1 — Re-checkpoint del audio al cambiar BPM:**

```ts
useEffect(() => {
  if (!audioRef.current || !bpmOriginalRef.current) return;
  const velocidad = Math.min(4, Math.max(0.1, bpm / bpmOriginalRef.current));
  audioRef.current.playbackRate = velocidad;
  (audioRef.current as any).preservesPitch = true;

  // RE-CHECKPOINT: alinear puntos de referencia
  if (reproduciendo && !pausado) {
    checkpointTimeRef.current = audioRef.current.currentTime;
    checkpointTickRef.current = tickActual;
  }
}, [bpm]);
```

**Fix 2 — Corrección suave continua con micro-modulación de `playbackRate`:**

En lugar de saltar el `currentTime` (clicks audibles), el sync loop modula `playbackRate` ±8% temporalmente cuando el drift es moderado (0.04s a 0.25s). Solo hace seek directo si el drift es enorme (>0.25s).

```ts
const diferencia = tiempoEsperado - tiempoActual; // signo importa

if (Math.abs(diferencia) > 0.25 && ahoraMs - lastSync > 1500) {
  audio.currentTime = tiempoEsperado;            // seek directo (raro)
  audio.playbackRate = velocidadBase;
  // re-checkpoint
} else if (Math.abs(diferencia) > 0.04) {
  const factorCorreccion = diferencia > 0 ? 1.08 : 0.92;
  audio.playbackRate = velocidadBase * factorCorreccion;  // corrección suave
} else {
  audio.playbackRate = velocidadBase;            // sincronizado
}
```

Resultado: el MP3, las notas y los clicks se mantienen pegados aunque se cambie el BPM múltiples veces durante minutos.

### Fix CRÍTICO — Pantalla roja al jugar secciones intermedias

**Síntoma:** al jugar la sección "Intermedio Primavera" (tick 38671) la vida caía a 0 instantáneamente y la pantalla se ponía roja.

**Causa raíz:** el monitor de ticks (línea 355 de `useLogicaProMax.ts`) iteraba sobre **toda** `cancionSeleccionada.secuencia` y marcaba como `'perdida'` cualquier nota cuyo `tick + ventana < tickActual`. **Fix:** filtrar el monitor al rango activo de la sección:
```ts
const seccionActiva = seccionSeleccionadaRef.current;
const rangoMin = seccionActiva ? seccionActiva.tickInicio - ventanaTicks : -Infinity;
const rangoMax = seccionActiva ? seccionActiva.tickFin + ventanaTicks : Infinity;

for (const nota of cancionSeleccionada.secuencia) {
  if (nota.tick < rangoMin || nota.tick > rangoMax) continue;
  // ... lógica original
}
```

## 37. Sincronización MP3 ↔ Secuencia — bugs y soluciones acumuladas

Este fue el problema más persistente de la sesión. Pasó por **5 iteraciones** hasta resolverse:

### Iteración 1 — Sync runtime con `setInterval`

Primer intento: cada 250ms comparar `audio.currentTime * factor` vs `tickActual` y llamar `sincronizarRelojConPista(tickReal)` si el drift > 80ms. **Falló:** los hard-syncs causaban saltos audibles.

### Iteración 2 — Pre-load + seek antes del conteo

El usuario notó: "la introducción funciona porque tiene un conteo de 3s, pero las otras secciones no". La introducción (tick≈0) no necesita seek; las demás sí.

**Fix:** crear el audio + setear `audio.currentTime = offsetSegundos` **inmediatamente** al pulsar Empezar. Promise `audioListo` que resuelve cuando dispara `'seeked'`. Conteo de 3s corre en paralelo (modo competencia). Al terminar el conteo, `audio.play()` + `reproducirSecuencia()` simultáneos en el mismo `requestAnimationFrame`. **Mejora parcial** pero el desfase persistía.

### Iteración 3 — Reloj sigue al `audio.currentTime`

Diagnóstico final: el reproductor usaba `motorAudioPro.tiempoActual` (AudioContext) mientras el MP3 usa `HTMLMediaElement.currentTime` — **dos relojes independientes**. Por eso la grabación quirúrgica funcionaba (RAF en `performance.now()`, mismo wall clock que el audio) pero el simulador no.

**Fix:** método nuevo `setAudioSync(audio, bpmOriginal)` en `useReproductorHero`:
```ts
const audioSyncRef = useRef<HTMLAudioElement | null>(null);
const bpmOriginalSyncRef = useRef<number>(120);

const setAudioSync = useCallback((audio, bpmOriginal?) => {
  audioSyncRef.current = audio;
  if (typeof bpmOriginal === 'number') bpmOriginalSyncRef.current = bpmOriginal;
}, []);
```

En el RAF:
```ts
if (audioActivo) {
  // El tick refleja exactamente la posición del MP3
  nuevoTickAbsoluto = audioSync.currentTime * (bpmOriginalSyncRef.current / 60) * resolucion;
} else {
  // Fallback: AudioContext time (modo libre, sin mp3)
  ...
}
```

`useLogicaProMax.ts` registra el audio antes de `dispararJuegoSincronizado`. Misma técnica aplicada al RAF de `ModalEditorSecuencia.tsx`. **Mejora grande** pero todavía había desfase al INICIO.

### Iteración 4 — Eliminar sync runtime redundante

Como el RAF ahora lee `audio.currentTime` cada frame, el `setInterval` de 250ms sobraba. Removido. **Sin mejora** (no era el problema real).

### Iteración 5 — DEFINITIVO: bloquear RAF hasta que audio dispare `'playing'`

**Diagnóstico final:** `audio.play()` es asíncrono. El audio tarda 100-300ms en producir sonido (especialmente con offset). Durante ese tiempo:
- Reloj de ticks: avanzando con `performance.now()` → dispara las primeras notas.
- Audio MP3: aún cargando/buffering → silencio total.
- `notasYaSonadasRef` impide que se vuelvan a disparar cuando el audio finalmente arranca.

**Resultado:** las notas suenan ANTES que el MP3 (que era exactamente lo que reportaba el usuario: "siempre inicia la pista primero y el mp3 después").

**Fix definitivo:**

```ts
// Editor admin — ModalEditorSecuencia.tsx
const audioRealmenteSonandoRef = useRef<boolean>(false);

// En el RAF:
const audioActivo = audioRealmenteSonandoRef.current
  && !!(audio && !audio.paused && audio.readyState >= 2);

if (audioActivo) {
  newTick = audio.currentTime * (bpmOrig / 60) * resolucion;
} else if (hayAudio) {
  // Audio pendiente de arrancar — CONGELAR el reloj
  newTick = tickLocalRef.current;
} else {
  // Sin audio: performance.now()
  newTick = ...;
}

// Listeners permanentes
audio.addEventListener('playing', () => { audioRealmenteSonandoRef.current = true; });
audio.addEventListener('seeked', () => { if (!audio.paused) audioRealmenteSonandoRef.current = true; });
audio.addEventListener('pause', () => { audioRealmenteSonandoRef.current = false; });

// Reset al pulsar Play / hacer seek
// Fallback de 1500ms si 'playing' no llega
```

**Misma técnica en el simulador** — `audioSyncSonandoRef` interno a `useReproductorHero`. `setAudioSync` registra los listeners automáticamente al recibir el audio. El RAF tiene 3 ramas:
1. Audio sonando → tick = `audio.currentTime * factor`
2. Audio registrado pero pendiente → tick **CONGELADO**
3. Sin audio → AudioContext time

Esto garantiza que las notas **NUNCA** se disparen antes que el MP3. Si el audio aún no produce sonido, el reloj literalmente se queda esperando.
